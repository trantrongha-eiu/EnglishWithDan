# Maintenance

Versioning, dependency-update policy, and — the most valuable part of this doc — an honest list of known technical debt and limitations. Every item below was found during real work on this codebase (not invented for completeness), so each one is real and worth reading before you assume a piece of the system is more polished than it is.

## Versioning strategy

There is no formal semver/release-tagging scheme in this repo today (no `CHANGELOG.md`, no git tags at the time of writing). Deploys are continuous — every push to `main` that passes CI and lands on the tracked branch auto-deploys via Render (see [`docs/DEPLOYMENT.md`](DEPLOYMENT.md)). This is a reasonable model for a single-team, continuously-deployed app of this size; if the team grows or external integrators start depending on the API, revisit this — at minimum, tagging releases and keeping a changelog would make [`docs/RELEASE_CHECKLIST.md`](RELEASE_CHECKLIST.md)'s "what changed" step something you can point to instead of reconstructing from commit history.

## Dependency updates

- Run `npm audit` in `backend/`, `admin-src/`, and `frontend/` periodically — not just when something feels wrong. A production-readiness audit found 12 known CVEs across backend dependencies that had accumulated silently; 11 were safe, non-breaking patches applied in one pass.
- **`nodemailer` has open HIGH-severity CVEs** (SMTP/CRLF injection, among others) at the currently-installed version (6.x). Fixing requires a breaking major-version upgrade (to 9.x) that was deliberately not auto-applied — the safety-check in this engagement's tooling specifically flagged it as needing human review of the email-sending code paths (password-reset OTP, writing-graded notifications) before upgrading. This is the single highest-priority dependency item outstanding.
- Prefer patch/minor updates freely; treat any major-version bump as its own reviewed change (re-read the package's changelog for breaking changes, re-run the full test suite, manually smoke-test the affected feature) rather than bundling it into an unrelated PR.
- `express-mongo-sanitize` is listed as a backend dependency but is not actually used anywhere in the code — the app has its own hand-rolled `backend/middleware/mongoSanitize.js` instead (written to work around an Express 5 compatibility issue the third-party package doesn't handle). Safe to remove from `package.json` next time you're touching dependencies; not urgent on its own.

## Known technical debt

Ranked roughly by how much it should influence a decision to fix it (highest-leverage first), not by discovery order.

1. **Secrets leaked via git history, still unrotated.** `JWT_SECRET`, `MONGO_URI`, `CLOUDINARY_API_SECRET` — see [`docs/SECURITY.md`](SECURITY.md) for the full picture. This is not "debt" in the usual sense — it's a live risk — but it belongs at the top of any list of things to fix regardless of what else is being worked on.
2. **`POST /api/task2template/save` trusts client-supplied grading.** Found while writing [`docs/API.md`](API.md). Unlike every other practice-save endpoint, this route (`task2TemplateService`/`Task2TemplateAttempt` — a separate service/model pair from `task2PracticeService`/`Task2Attempt`) only coerces the client-sent `totalItems`/`correctItems` with `Number()` rather than re-deriving them from the real answer key. A client can submit a fabricated score directly. Same class of risk as the server-side re-grading fix already applied elsewhere, but this endpoint fell outside that fix's original scope. See [`docs/SECURITY.md`](SECURITY.md) for the full disclosure. Worth a targeted fix in a future phase.
3. **`nodemailer` breaking-change upgrade pending.** See above.
4. **`User` deletion doesn't cascade.** Found while writing [`docs/DATABASE.md`](DATABASE.md). Deleting a `User` document does not clean up the roughly 15 other collections that reference it via `userId`/`studentId` (attempts, vocab books, drafts, messages, etc.) — those documents become orphaned, referencing a user that no longer exists. Not currently causing visible bugs (nothing crashes on a missing referenced user), but it means account deletion doesn't actually free the data it looks like it frees, and any future admin-facing "user data export/delete" (GDPR-style) feature would need to account for this explicitly rather than assuming a single delete is sufficient.
5. **Structured logging is partially adopted.** `backend/utils/logger.js` covers the specific high-value events the production-readiness phase targeted (startup/shutdown/security/auth/database/AI failures) — roughly 70 pre-existing `console.log`/`console.error` calls elsewhere in the codebase were deliberately left as-is (touching all of them was judged disproportionate churn for that phase's scope). Migrating the rest — especially `backend/middleware/catchAsync.js`'s `console.error`, which is the single choke point most routes actually funnel errors through — would make log search genuinely complete rather than "mostly complete." Good first-week task for a new backend contributor: pick one file, migrate its logging, see the pattern.
6. **`AppError` typed-error hierarchy is opt-in, not the norm.** `backend/errors/AppError.js` exists and a few routes use it, but the large majority of controllers still do manual `try/catch` + `res.status(...).json({success:false, message:err.message})`. Both patterns coexist correctly today (the global `errorHandler.js` handles either), but a future contributor reading two different controllers will see two different styles. Recommend the `AppError` pattern for new code; retrofitting old code is optional cleanup, not a bug.
7. **`AccessKey` feature appears orphaned.** The model and admin CRUD UI for generating access keys exist (`backend/models/AccessKey.js`, `backend/routes/admin/accessKeys.js`), but no student-facing "redeem a key" endpoint exists anywhere in the codebase — `currentUses` is defined on the schema but nothing ever increments it. This looks like a feature that was superseded by the `plan`/`requirePremium` gating system and never fully removed. Worth a product decision: finish it, or delete the dead code path.
8. **`WPLesson` model appears write-only.** Found while writing [`docs/DATABASE.md`](DATABASE.md). Admin routes create/update `WPLesson` documents, but no read path (student-facing or admin) was found anywhere in the codebase that queries them back out. Same shape of issue as `AccessKey` above — likely a feature that was started and never finished, or superseded before shipping. Worth a product decision alongside item 7 rather than fixing in isolation.
9. **Orphaned Cloudinary assets on content deletion.** Deleting a `Passage`/`ReadingTest`/`ListeningTest` does not delete its associated Cloudinary media (image/audio) — storage accumulates indefinitely with no cleanup job. Not a security issue, a cost/hygiene one. See [`docs/BACKUP_RESTORE.md`](BACKUP_RESTORE.md) §3.
10. **`task2/draft`'s `saveDraft()` may silently delete drafts for other topics — needs intent confirmation.** Found while writing [`docs/API.md`](API.md). Saving a draft for one Task 2 topic appears to delete the user's saved drafts for every *other* topic in the same call, despite a plural `GET /api/task2/drafts` endpoint existing that implies multiple concurrent drafts are a supported concept. This wasn't confirmed as unintentional (it may be deliberate "one active draft at a time" behavior), but the mismatch with the plural list endpoint is suspicious enough to flag rather than silently leave undocumented. Needs a product-owner read before treating it as a bug to fix.
11. **No shared `Table`/list component in the admin SPA.** Every admin page that shows a data table (Users, Attempts, Passages, etc.) builds its own inline — there's no `<Table>`/`<DataGrid>` abstraction, even though the shape is repeated across ~15 pages. `Pagination` *is* extracted as a shared component; a table wrapper isn't. Worth extracting once a third or fourth admin page needs the same table-with-sort-and-filter shape, not preemptively.
12. **Mongoose `{new: true}` deprecation warning.** A recent mongoose version bump (part of the CVE patching above) surfaced a real, currently-harmless deprecation warning: the `{new: true}` option on `findOneAndUpdate`/`findOneAndReplace` should become `{returnDocument: 'after'}`. Used across several call sites (e.g. `backend/routes/admin/users.js`). Low priority — a future mongoose major version may remove the old option entirely, so it's worth a mechanical find-and-replace + re-test pass eventually, just not urgently.
13. **`config/index.js`'s centralization is partial.** It's described as centralizing environment-variable access, but roughly a dozen files still read `process.env.X` directly instead of going through it (`middleware/auth.js`, `services/geminiService.js`, several controllers). Not a bug — `dotenv` loads before any of these run either way — but it means a future rename of an env var needs a full-repo grep, not just an edit to `config/index.js`.
14. **`login.html`/`register.html` hardcode the production API URL.** Rather than reading a configurable API base URL, these two pages have `https://englishwithdan.onrender.com` written directly into their JS. This makes fully-local end-to-end testing of the login/register flow against a local backend impossible without a temporary manual edit. Every other frontend page routes through the shared `apiFetch`, which has the same underlying issue but is at least centralized in one place (`js/shared/api-client.js`) rather than duplicated per-page.
15. **`task1Practice.js` route comment is stale/incorrect.** Found while writing [`docs/API.md`](API.md). A route comment claims auth is optional on this route, but the route actually applies `auth` middleware unconditionally — the comment describes behavior that no longer matches the code. Low-risk, doc-only fix (correct or remove the comment); flagged here rather than fixed silently during a documentation-only pass, consistent with how the other comment/doc fixes in this phase were handled.

## Future roadmap

Nothing formally committed — this section exists so ideas that came up during the hardening engagement aren't lost, not as a promise of what's next. In rough priority order if the team is looking for what to tackle after the technical debt above:

- A real staging environment (currently: local dev, then straight to production — `docs/DEPLOYMENT.md` notes there's no staging-compatibility story beyond "the same Render setup, a different service").
- Finish or remove the `AccessKey` feature (see technical debt #5).
- A cleanup job for orphaned Cloudinary assets (see technical debt #6).
- Extend server-side re-grading to cover AI-graded question types too, if a way to do it without re-calling Gemini on every save can be found (e.g. caching the AI grading result server-side at first-check time and validating the client didn't tamper with which cached result it's claiming).
- A shared `Table` component for the admin SPA once the duplication pain is real (see technical debt #7).

## Known limitations

Distinct from technical debt (things that could reasonably be fixed) — these are current, deliberate, or structural boundaries of the system as it exists today:

- **No refresh-token mechanism.** A JWT is valid for exactly 7 days; there's no silent-refresh flow. A user's session simply expires and they log in again. Fine for this app's usage pattern; would need addressing if session length requirements change.
- **AI grading has no offline/local fallback.** If Gemini is down or misconfigured, essay/speaking/Task-2-sentence AI grading is unavailable (the app degrades gracefully — a friendly "try again" message, not a crash — but the feature itself has no substitute).
- **No staging environment** (see roadmap above) — every deploy to `main` goes to production after CI passes.
- **Single-region deployment.** No documented multi-region or failover setup; an Atlas/Render regional outage is a real outage for all users, not a partial-degradation scenario.
- **No admin audit-log UI.** Admin actions (role changes, bans, deletions) are now logged (`logger.security`, added in the production-readiness phase) but there's no admin-facing screen to browse that history — it's log-viewer-only today.
